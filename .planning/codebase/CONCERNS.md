## Tech Debt

### Smoke Test Failure — Empty Snapshots

**Issue:** Test results show `snapshots: []` and `result: FAIL` for smoke tests, indicating measurement failure rather than call failure.

**Files:** `backend/src/services/sippManager.js` (lines 134-184)

**Root Cause:** `buildSummary()` (line 201) relies on `parseSippStats()` (line 192) which parses stderr output with regex pattern `/(\d+)\s+calls/i`. With `stdio: ['ignore', 'pipe', 'pipe']` on line 161, stderr may be incomplete or lost, especially for short-duration tests. The 1-call smoke test (30 seconds) completes too quickly to accumulate meaningful snapshots.

**Impact:** Cannot verify that SIPp is executing correctly or gathering call metrics. Blocks validation of test infrastructure and prevents real load tests from proceeding with confidence.

**Fix Approach:** Read final `_statistics.csv` file generated by SIPp instead of parsing stderr. SIPp writes comprehensive statistics CSV at test completion. Parse that instead of ephemeral stderr output:
- After `sippProcess.on('close')` fires, read the `_statistics.csv` file from SIPp's output directory
- Extract call counts, response codes, timing, and success metrics from CSV headers
- Use those definitive values in `buildSummary()` instead of relying on stderr line fragments

### Parser Stale Warning — Undiagnosed

**Issue:** LogReader emits warning `[LogReader] No log data for 2+ minutes — parser may be broken` but root cause unknown.

**Files:** `backend/src/services/logReader.js` (lines 100-114)

**Cause:** The warning fires when `state.lastParsedAt` has not been updated for 2+ minutes (PARSER_STALE_MS = 120s). This could mean:
1. SSH stream disconnected and `attachStream()` reconnect failed silently
2. 3CX logs have no new events (low-traffic environment) — false positive
3. SSH data is arriving but `parseLine()` returns null for all lines (parser regex no longer matches 3CX log format after version update)
4. Encoding issue on tail output (binary/UTF-8 mismatch)

**Impact:** False alarms causing confusion; masks real parser breakage. If logs genuinely stopped parsing, users won't know until performance degrades.

**Fix Approach:**
1. Distinguish between "no events" (normal in low traffic) vs "stream broken" (reconnect watch)
2. Add per-log-type counters in `parseLine()` to track whether regex matches are happening at all
3. If a log file receives new tail output but produces zero events for 30+ seconds, alert separately as `parser_regex_broken`
4. Check SSH reconnect status and emit SSH-layer alert separately

### Destination Parameter Not Sanitized

**Issue:** SIPp destination (extension to call) passed directly from frontend without validation, hereditary from earlier phases.

**Files:** `backend/src/services/sippManager.js` (lines 56, 142)

**Current State:** Destination is used in SIPp `-s` flag (line 142). Although indirectly via `spawn()` argument array (safer than shell string), no validation exists for format.

**Risk:** Minimal (spawn with array is safe), but violates defense-in-depth. A malformed extension like `../etc/passwd` or `999\nmalicious_command` could cause SIPp to misbehave or fail cryptically.

**Fix Approach:**
- Validate destination is alphanumeric + underscore only: `/^[a-zA-Z0-9_]+$/`
- Reject with clear error message if invalid
- Store allowed extensions in `.env` or fetch from 3CX API

### Missing Input Validation on Test Parameters

**Issue:** While backend applies hard limits (lines 9-13), no validation for invalid types or negative numbers.

**Files:** `backend/src/services/sippManager.js` (lines 44-57, 50-55)

**Example:** `parseInt(preset?.calls ?? params.max_calls ?? 10)` will coerce `null`, `undefined`, or non-numeric strings to `NaN`, then `Math.min(NaN, limit)` returns `NaN`.

**Impact:** Frontend can send malformed data that silently defaults to incorrect values, causing tests to run with wrong parameters.

**Fix Approach:**
```javascript
const max_calls = parseInt(preset?.calls ?? params.max_calls ?? 10);
if (!Number.isInteger(max_calls) || max_calls < 1 || max_calls > LIMITS.maxCalls) {
  throw new Error(`Invalid max_calls: must be 1-${LIMITS.maxCalls}`);
}
```

## Security Concerns

### No Authentication or Authorization

**Issue:** Backend has no auth layer. Anyone on the network can invoke `/api/tests/run` and start SIPp tests.

**Files:** `backend/src/routes/tests.js` (lines 14-22), `backend/src/server.js` (lines 20)

**Attack Scenario:** Malicious actor repeatedly calls `/api/tests/run` with 200-call parameters, consuming CPU and bandwidth, making the 3CX unreachable or triggering SIP bans from carrier.

**Current Mitigation:** Single-test lock (`if (currentTest)` on line 45) prevents concurrent tests. Insufficient against sequential abuse.

**Fix Approach:**
- Add JWT or API key validation on test-creation endpoints
- Implement IP allowlist (restrict to known ops staff IPs)
- Add rate limiting per IP: max 1 test per 30 minutes per IP
- Log all test initiations with clear IP attribution (already done on line 16, but not exposed publicly)

---

### CORS Allows Any Origin

**Issue:** Backend CORS is `{ origin: '*' }` (server.js line 20).

**Files:** `backend/src/server.js` (line 20)

**Risk:** Frontend at different domain can make requests and WebSocket connections. If OLAM frontend is compromised, cross-origin JavaScript can abuse backend APIs.

**Fix Approach:**
- Set `origin` to specific frontend URL: `{ origin: process.env.FRONTEND_URL || 'http://localhost:5173' }`
- In production, restrict to 172.18.164.35 only

---

### SSH Key Management

**Issue:** Private SSH key stored unencrypted on disk at `backend/keys/3cx_rsa`. No documentation on key rotation or access control.

**Files:** `backend/src/services/sshClient.js` (line 33)

**Risk:** If deployment host is compromised, attacker gains SSH access to 3CX server with root privileges.

**Current Mitigation:** Git ignores `backend/keys/` (presumably). Manual key placement required.

**Fix Approach:**
- Document key placement and permissions (600 on `3cx_rsa`)
- For production: use SSH agent or environment-based auth (AWS Secrets Manager, HashiCorp Vault)
- Rotate key quarterly

## Performance Bottlenecks

### Node Exporter Scraping Blocking Metrics Collection

**Issue:** `fetchRealMetrics()` does HTTP GET to node_exporter with 4-second timeout (line 180).

**Files:** `backend/src/services/metricsCollector.js` (lines 175-207)

**Impact:** If node_exporter is slow or unreachable, metrics collection stalls for 4 seconds per poll cycle (typically 5 seconds). If carrier SIP errors spike during that gap, they're not visible.

**Fix Approach:**
- Move node_exporter scraping to separate background worker
- Cache last-known metrics for up to 30 seconds if scrape fails
- Do not block WebSocket updates on prometheus fetch latency

---

### LogReader SSH Stream Can Stall

**Issue:** `tail -F` over SSH can buffer output. `stdbuf -oL` mitigates (line 47) but single stream for 5 log files means any one slow file blocks the others.

**Files:** `backend/src/services/logReader.js` (lines 42-65)

**Impact:** If 3CXSystemService.log is being written heavily, CallFlow.log events may be delayed 10+ seconds.

**Fix Approach:**
- Open 5 separate SSH streams, one per log file
- Merge events in order by timestamp in a small in-memory queue
- Trade CPU/memory for responsiveness

## Fragile Areas

### Log Parser Brittle to 3CX Version Changes

**Issue:** Parser uses hardcoded regex patterns (lines 32-66 in logParser.js) tuned to 3CX v20 Update 8 Build 1121. Any version update may change log format.

**Files:** `backend/src/services/logParser.js`

**Examples of Fragility:**
- Line 32: `TS_YMD_RE` expects `YYYY/MM/DD HH:MM:SS.mmm` but 3CX could change to ISO 8601
- Line 53: `CALL_INVITE_RE` looks for `Call[Ii]d|callId` but 3CX could rename to `SessionId`
- Line 58: `ERR_408_RE` matches `SIP/2.0 408` but text logs might omit version

**Risk:** 3CX auto-updates are enabled (finding H-05). Silent update breaks parser; warning fires after 2 minutes of data loss.

**Fix Approach:**
- Add schematic version detection: query 3CX API at startup to confirm log format
- If format changed, emit critical alert with recommendation to disable auto-updates
- Provide fallback "lenient" parser that extracts only call counts and errors, ignoring format subtleties
- Document expected log lines with examples; maintain test fixture of real logs from each version

## Known Bugs

### UTF-8 to Latin-1 Encoding Corruption in Alerts

**Issue:** Alerts display corrupted accented characters (e.g., `Ã¡` instead of `á` in Spanish text).

**Files:** Likely in `backend/src/services/logReader.js` (line 87) or frontend rendering

**Example:** Alert message `"Troncal Tigo UNE desregistrada"` renders as `"Troncal Tigo UNE desregistrada"` (visually similar but bytes wrong).

**Cause:** Mismatch between file encoding and HTTP response/WebSocket encoding. SSH stream from Debian may send UTF-8; JavaScript may be assuming Latin-1.

**Fix Approach:**
- Ensure `.env` specifies UTF-8 locale on Debian side: `LC_ALL=C.UTF-8`
- Frontend: verify `<meta charset="utf-8">` in HTML
- Backend: explicitly `.toString('utf8')` on all chunk data (logReader.js line 53)

### disk.recordings Always 0

**Issue:** Recordings disk usage always reported as 0 in metrics.

**Files:** `backend/src/services/metricsCollector.js` (lines 189-199)

**Root Cause:** No separate mount for recordings on 3CX. Code looks for `/var/lib/3cxpbx/Instance1/Recordings` mount (line 191) but it doesn't exist; falls back to 0 (line 199).

**Impact:** Dashboard metric is useless; no early warning if recordings fill disk. For long soak tests, could cause call drops.

- Document in deployment guide that recordings should use dedicated volume for monitoring to work

### MOS, Jitter, Packet Loss Not Collected

**Issue:** Quality metrics (MOS, jitter, packet loss) are hardcoded to 0 in real mode (metricsCollector.js lines 220-225).

**Files:** `backend/src/services/metricsCollector.js` (lines 220-225)

**Why:** These require RTCP packets from RTP stream. 3CX must be configured to record RTCP, and backend must parse them. Not yet implemented.

**Impact:** Quality tab on dashboard shows no data. Cannot detect voice degradation (jitter spike, packet loss > 1%).

- If monitoring production traffic: install SIPp's RTP proxy to mirror packets, extract RTCP
- Alternative: hook into 3CX Call Control API's recording API to fetch call QoS metrics post-call
- Document as "placeholder" in KPI table with note "Requires RTCP instrumentation"

## Scaling Limits

### Single SSH Connection to 3CX

**Issue:** All log reading and commands go through one persistent SSH connection.

**Files:** `backend/src/services/sshClient.js` (lines 20-60)

**Limit:** If connection is saturated by `node_exporter` requests or large log volumes, subsequent command execution (e.g., `tcpdump` for capturing RTP) will be delayed.

**Current Band-Aid:** Tunnel for node_exporter (lines 62-97) offloads HTTP scraping to separate TCP tunnel.

**Fix Approach:**
- Open 2nd SSH connection specifically for metrics (node_exporter via SSH tunnel)
- Keep 1st connection for log streaming only
- Keep 3rd connection pool (2-3 connections) for ad-hoc commands (tcpdump, configuration reads)

---

### In-Memory Alert Deduplication

**Issue:** `lastFiredAt` Map in anomalyDetector.js (line 100) is not persisted. Restart loses history; same alert fires immediately on reboot.

**Files:** `backend/src/services/anomalyDetector.js` (lines 100-127)

**Impact:** If backend crashes and restarts, users see duplicate alerts for conditions that were already firing (alert fatigue).

**Fix Approach:**
- Persist `lastFiredAt` to SQLite keyed by alert ID and timestamp
- Load at startup
- Garbage-collect entries > 24 hours old
