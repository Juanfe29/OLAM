### 2. Auditoría de bloqueos para pruebas reales

Se validó el estado real de la config del backend:
- ❌ `backend/.env` no existe (solo está `.env.example`).
- ❌ `backend/keys/3cx_rsa` no existe.
- ❌ SIPp no está instalado en esta máquina (`which sipp` falla).
- ❌ No se validó acceso de red al 3CX (requiere VPN Fortinet).

Se identificaron bloqueos en tres capas:

#### SSH / logs del 3CX
1. Crear `backend/.env` con `MOCK_MODE=false`.
2. Colocar clave privada en `backend/keys/3cx_rsa` (chmod 600).
3. Validar acceso SSH manual antes de arrancar el backend: `ssh -i backend/keys/3cx_rsa root@172.18.164.28 "ls /var/lib/3cxpbx/Instance1/Data/Logs"`.
4. Conectarse a la VPN Fortinet para alcanzar la red `172.18.164.0/24`.

#### Prometheus / node_exporter
1. Validar que `node_exporter` está corriendo en el 3CX: `curl http://172.18.164.28:9100/metrics`.
2. Si no está instalado: `apt install prometheus-node-exporter && systemctl enable --now prometheus-node-exporter` en el 3CX.
3. **Problema latente:** [metricsCollector.js:176-178](../backend/src/services/metricsCollector.js#L176-L178) devuelve todos ceros cuando falla la conexión a node_exporter, **en silencio**. Un assessment con dashboard en 0% CPU es indistinguible de "servidor no responde". Hay que hacer el error visible.
4. La partición de grabaciones nunca se monitorea ([metricsCollector.js:170](../backend/src/services/metricsCollector.js#L170) hardcodea `recordings: 0`).

#### SIPp
1. Instalar SIPp v3.7.7 en el host del backend (**nunca en el 3CX** — invalida métricas).
   - Linux: `apt install sip-tester` o compilar desde [github.com/SIPp/sipp](https://github.com/SIPp/sipp).
   - Windows: binario precompilado, pero performance cae >50 calls/s.
   - **Recomendación:** correr el backend desde VM Linux si se van a hacer pruebas >100 concurrentes.
2. El target está hardcodeado a puerto 5060 UDP sin TLS ([sippManager.js:135](../backend/src/services/sippManager.js#L135)) — coincide con el hallazgo H-07 del assessment.
3. **Vulnerabilidad viva:** el campo `destination` se pasa al comando SIPp sin sanitizar ([sippManager.js:142](../backend/src/services/sippManager.js#L142)). Riesgo de inyección de shell desde el frontend.
4. El parser de stats SIPp solo captura conteo total de llamadas ([sippManager.js:185-192](../backend/src/services/sippManager.js#L185-L192)) — no extrae PDD, ASR ni MOS reales del proceso.

### 3. Decisión de arquitectura SSH — Opción B

Se evaluaron dos caminos para conectar el backend al 3CX:

| Opción | Descripción | Decisión |
|---|---|---|
| **A** | Parchear `sshClient.js` para aceptar `SSH_PASSWORD` además de `SSH_KEY_PATH`. | ❌ Descartada |
| **B** | Generar par de claves local → ssh-copy-id al 3CX con password una sola vez → backend usa la clave privada de ahí en adelante. | ✅ **Elegida** |

**Motivación:** la opción B es más segura (sin passwords en `.env`), más robusta ante rotación de password root, y solo requiere usar el password una sola vez (para instalar la pública).

**Implementación pendiente (próxima iteración):**
```bash
# 1. Generar par de claves (ed25519, sin passphrase para el backend)
ssh-keygen -t ed25519 -f backend/keys/3cx_rsa -N ""

# 2. Conectarse a VPN Fortinet primero

# 3. Instalar pública en el 3CX (pide el password root UNA vez)
ssh-copy-id -i backend/keys/3cx_rsa.pub root@172.18.164.28
# — alternativa manual si ssh-copy-id no está disponible:
#   cat backend/keys/3cx_rsa.pub | ssh root@172.18.164.28 "cat >> ~/.ssh/authorized_keys"

# 4. Verificar que entra sin password
ssh -i backend/keys/3cx_rsa root@172.18.164.28 "hostname"

# 5. Crear backend/.env con MOCK_MODE=false y SSH_KEY_PATH=./keys/3cx_rsa
```
