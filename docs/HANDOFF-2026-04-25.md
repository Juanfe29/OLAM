### 2. node_exporter + SSH tunnel para Prometheus

**Lo encontrado:**
- node_exporter v1.5.0 ya estaba instalado en el 3CX, solo no estaba enabled.
- `apt install -y prometheus-node-exporter && systemctl enable --now prometheus-node-exporter` lo activó.
- ✅ Servicio escuchando en `*:9100`.

**Sorpresa:** firewall **upstream** (router OLAM o policy Fortinet) bloquea el puerto desde la subnet VPN. Confirmado con tcpdump en el 3CX: SYN packets nunca llegan al server. El firewall del 3CX (nftables) tiene `tcp dport 9100 accept` correctamente.

**Solución aplicada:** SSH tunnel en [sshClient.js](../backend/src/services/sshClient.js) — `forwardOut('127.0.0.1', 0, '127.0.0.1', 9100, ...)`. El backend pega a `http://127.0.0.1:9100` y el túnel forwardea por la conexión SSH ya existente.

**Detalle técnico:** los listeners de `error` deben estar **antes** de `socket.pipe(stream).pipe(socket)` — sino un `ECONNRESET` en cualquiera de las puntas tira el proceso. Aprendido a la mala (crash de backend hace unas horas).

**Cuando OLAM abra el puerto upstream:** poner `NODE_EXPORTER_VIA_SSH=false` en `.env` y `NODE_EXPORTER_URL=http://172.18.164.28:9100/metrics`. El código sigue siendo válido.

### 3. Regex de host arreglados ([metricsCollector.js](../backend/src/services/metricsCollector.js))

### 4. Parser de logs — el bloqueo más grande del día

**Problemas encontrados:**

1. **Buffering de stdout** — `tail -F` con stdout pipeado por SSH bufferiza por bloques (4KB+). Líneas individuales se quedan en buffer indefinidamente. **Fix:** prepend `stdbuf -oL` al comando.
2. **Flag `-q`** — `tail -Fq` SUPRIME los headers `==> file <==`. Pero `detectFile()` los necesita para saber qué archivo es cada línea. **Fix:** quitar `-q`. Ahora el tail emite headers, el parser asigna el archivo correctamente.
3. **`-n 0`** — sin esto, cada arranque del backend dumpeaba todo el histórico de los 5 logs (MB de datos). Agregado para evitar.
4. **Regex de timestamp inválido** — esperaba `[DD/MM/YYYY HH:MM:SS.mmm]` con corchetes. El formato real:
   - 4 logs: `YYYY/MM/DD HH:MM:SS.mmm|####|Level|...` (CallFlow, Gateway, Queue, System).
   - IVR: `DD/MM/YYYY HH:MM:SS.mmm [hex] ...`.
   - **Ningún log usa corchetes**. **Fix:** dos regex (`TS_YMD_RE`, `TS_DMY_RE`).
5. **Falso positivo de 408/503** — regex `/408/` matcheaba cualquier "408" en cualquier parte (incluyendo timestamps tipo `04:08`). **Fix:** regex estricto exigiendo contexto SIP: `\b(SIP\/2\.0\s+408|408\s+Request\s+Timeout|status[:=]\s*408)\b`.
6. **SystemService — abreviación 3CX** — el log usa `Erro` (sin la `r` final), no `Error`. **Fix:** detectar nivel en formato pipe-separated `\|(Erro|Crit|Fatal|Warn)\|`.
7. **CallFlow regex inventados** — los ejemplos del comentario nunca aparecen en logs reales. Sin tráfico SIPp no podemos validar el formato exacto. **Fix temporal:** patrones defensivos basados en docs 3CX (INVITE, BYE, CANCEL, ParentConnectionTerminated, Released).
8. **State con Set sin callId estable** — `state.activeCalls = new Set()` requería un callId para hacer add/delete. Como las líneas reales no tienen callId estable, `delete(undefined)` no hacía nada. **Fix:** counter incremental con piso en 0.

**Validación end-to-end del fix:**

Inyecté 6 líneas SIP/2.0 408 vía SSH:
```
==> /var/lib/3cxpbx/Instance1/Data/Logs/3CXGatewayService.log <==
2026/04/25 15:10:00.000|9999|Info| SIP/2.0 408 single line test
2026/04/25 15:11:00.000|9999|Info| SIP/2.0 408 batch 1
... (4 más)
```

Resultado del API: `{"errors408": 6, "errors503": 0}` ✅

**Latencia observada:** ~3-5s entre escritura del log y conteo en backend. Es por buffering del SSH stream (poll de tail cada 1s + agrupamiento del canal SSH). Aceptable para dashboard de monitoreo.

## Detalles técnicos para próxima sesión

### Comando final del log reader
```js
const cmd = `stdbuf -oL tail -F -n 0 ${LOG_FILES.join(' ')} 2>&1`;
```

- `stdbuf -oL`: line-buffered (sino bloques 4KB).
- `tail -F`: follow + manejar rotaciones por nombre de archivo.
- **Sin `-q`**: los headers `==> file <==` SON necesarios para detectar archivo.
- `-n 0`: solo seguir lo nuevo, no dumpear histórico.

### SSH tunnel — patrón
```js
ssh.connection.forwardOut(
  '127.0.0.1', 0,
  '127.0.0.1', TUNNEL_REMOTE_PORT,
  (err, stream) => {
    if (err) { socket.destroy(); return; }
    // Listeners ANTES del pipe, sino ECONNRESET tira el proceso
    socket.on('error', () => stream.destroy());
    stream.on('error', () => socket.destroy());
    socket.pipe(stream).pipe(socket);
  },
);
```

### 8. ⚠️ Donde quedamos pegados: zombie processes + RDP logoff

**Pasó esto:**
1. Primer `npm start` arrancó OK, vimos `[OLAM] Backend running on http://localhost:3000 [PRODUCTION MODE]`.
2. Después de hacer robocopy de los archivos faltantes, intentamos `npm start` de nuevo → **EADDRINUSE :::3000**.
3. Investigamos los procesos: hay 3 nodes (PID 5012, 8256, 5544) en el puerto 3000. **Owner: lamda** (los nuestros), pero `Stop-Process` y `taskkill` los dos fallaron con "Acceso denegado".
4. Razón: los procesos zombies sobrevivieron de una sesión RDP anterior. Windows aísla sesiones y la sesión nueva no puede matar los procesos de la vieja sin admin.
5. Intentamos cerrar sesión RDP (Inicio → Cerrar sesión). El logoff quedó **pegado en la pantalla "Espera"** porque los procesos node no responden al signal de cierre.
6. Sin admin no podemos `shutdown /r /m \\172.18.164.35` (error 53: equipo no permite shutdown remoto).

**Estado al cierre de esta sesión:** equipo `172.18.164.35` quedó en pantalla azul "Espera". Esperando que el admin de OLAM reinicie físicamente (o desde el hipervisor si es VM).

**Lección para próximas veces:** **siempre Ctrl+C el backend antes de cerrar sesión RDP.** Los procesos node detached no responden al SIGTERM del logoff y dejan la sesión zombie. Cuando reconectás con la misma cuenta no-admin, los huérfanos no se pueden eliminar.

Mientras eso no esté, el modo pasivo es 100% funcional y representa la mayor parte del valor del assessment (dashboard, alertas, KPIs, anomaly detection, history). Las pruebas SIPp activas son bonus para validar capacidad de carga.

## Lecciones del día

1. **Path length de Windows + `Compress-Archive`** = pérdida silenciosa de archivos. Siempre validar con count o usar robocopy directo.
2. **`tsclient` solo aparece si "Unidades" está tildado** en RDP options ANTES de conectar. Tildar solo "Portapapeles" no alcanza.
3. **Procesos node detached + RDP logoff = deadlock**. Siempre Ctrl+C servidores antes de cerrar sesión.
4. **Sin admin en Windows**, `Stop-Process` y `taskkill` fallan en procesos de sesiones RDP anteriores aunque sean del mismo usuario.
5. **DNS bloqueado pero TCP abierto** es un patrón corporativo común. Bundlear todo (node_modules, runtime portable) es la única salida viable.
6. **Estar en LAN del 3CX simplifica radicalmente** la arquitectura — todos los workarounds de la sesión 1 (SSH tunnel, autorizar IP en peer SIP) se eliminan.
