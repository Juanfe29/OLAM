#### BLOCK-01 — Validación de extensión destino

**Approach:** lista estática vía `VALID_EXTENSIONS` en `.env` (CSV). Razón: zero latencia, fail-safe en formato, sin dependencia de API 3CX que está sin documentar. Trade-off aceptado: actualización manual cuando OLAM confirme nuevas extensiones.

**Endpoints/UI nuevos:**
- `GET /api/tests/destinations` → `{ valid: [...] }` con la lista
- `TestControl.jsx` muestra chips clickeables con extensiones válidas
- `POST /api/tests/run` rechaza con 400 + mensaje claro si la extensión no está en allowlist; `testId` nunca se crea

**Cambio adicional:** default `destination` en TestControl pasó de `'100'` a `'1910'` (la única ext con digest auth ya validada en commit `ac11d81`).

#### BLOCK-02 — Lectura de `_statistics.csv` final de SIPp

**Approach:** `chokidar` para watch + `fast-csv` para parsear. SIPp invocado con `cwd` dedicado en `os.tmpdir()` para encontrar el CSV. Espera hasta 8s con `awaitWriteFinish` (500ms grace). Fallback al parser de stderr si el CSV no aparece (no rompe invocaciones donde SIPp crashea antes de escribirlo).

**Función nueva:** `buildSummaryFromCsv(csv, params, fallbackSnapshots)` usa los cumulativos canónicos de SIPp (`TotalCallCreated`, `SuccessfulCall`, `FailedCall`, `CurrentCall`, `ResponseTime1Avg`).

**Flag operacional:** `summary.csvMissing = true` cuando se cae al fallback de stderr — útil para diagnosticar tests sospechosos.

#### BLOCK-03 — State machine del parser/SSH

**Cuatro estados posibles** evaluados cada 30s en `diagnoseParserState()`:

| Estado | Condición | Severidad alerta | Mensaje |
|---|---|---|---|
| `ssh_down` | `isConnected() === false` | CRITICO | "Conexión SSH al 3CX caída — sin métricas en vivo" |
| `parser_broken` | SSH up + líneas llegan + ninguna matchea regex >2min | ALTO | "Parser de logs sin matches Xmin — formato puede haber cambiado" |
| `no_traffic` | SSH up + no llegan líneas crudas >2min | BAJO | "Sin tráfico real, esperado fuera de horario" |
| `ok` | resto | (limpia alertas) | — |

**Cambios estructurales:**
- `state.linesReceived` y `state.linesMatched` (contadores nuevos en logReader)
- Cooldown 60s para no spamear el WebSocket (snapshot persiste igual via `addExternalAlert`)
- `addExternalAlert(alert)` y `clearExternalAlert(id)` exportados en `anomalyDetector.js` para que el snapshot devuelto por `GET /api/status` incluya estas alertas (antes se perdían si el frontend reconectaba)

**AlertPanel.jsx no requiere cambios** — ya soporta CRITICO/ALTO/MEDIO/BAJO via `LevelBadge` en `StatusBadge.jsx`.
