## Lo que se hizo en esta sesión

### 1. Switch de target a `.33` (SC256)

OLAM activó un server 3CX trial con licencia SC256 en `172.18.164.33` específicamente para tests de carga >32 calls. El `.env` se actualizó:

```env
SSH_HOST=172.18.164.33
SSH_PORT=22
SSH_USER=root
SSH_PASSWORD=Olam2026$
```

**Confirmado**: el `.33` es **independiente** del `.28`. No comparten base de extensiones ni passwords SIP. Esto se descubrió empíricamente — más detalle en sección "Bloqueante".

### 2. Fix #1: `-t u1` y `-recv_timeout`

**Problema original:** tests de 30+ calls fallaban con "UDP retransmission timeout" porque SIPp envía INVITEs con retransmisiones agresivas y el 3CX no respondía a tiempo.

**Fix aplicado:** ([sippManager.js:117-118](../backend/src/services/sippManager.js#L117-L118))
```js
'-t',  'u1',                  // UDP single socket
'-recv_timeout', '15000',     // 15s para que el 3CX bajo carga tenga tiempo de responder
```

**Importante:** intentamos `-t un` (UDP socket per call) pero **rompe en Cygwin/Windows**:
```
Maximum number of open sockets (50000) should be less than the maximum number of open files (3200).
```
SIPp pre-aloca 50000 socket slots y el ulimit de Cygwin es 3200. Volvimos a `-t u1` definitivamente.

### 3. Fix #2: Scenario XML con auth digest

**Problema:** el built-in `-sn uac` no maneja el `SIP/2.0 407 Proxy Authentication Required` que devuelve el 3CX. Aunque pasáramos `-au <user> -ap <pass>`, el built-in los ignora — esos flags solo aplican a scenarios XML con placeholder `[authentication]`.

**Fix aplicado:**

- **Nuevo archivo:** [`backend/sipp-scenarios/uac_auth.xml`](../backend/sipp-scenarios/uac_auth.xml)
  - Hace el handshake completo: INVITE → 407 → ACK → INVITE+`[authentication]` → 100/180/200 → ACK → pause → BYE → 200
  - Usa `auth="true"` en el `<recv response="407">` para capturar el challenge

- **`sippManager.js` modificado:** ([sippManager.js:106-122](../backend/src/services/sippManager.js#L106-L122))
  - Si `SIPP_AUTH_USER` y `SIPP_AUTH_PASS` están en `.env` → usa `-sf <ruta>/uac_auth.xml`
  - Si no, fallback a `-sn uac`

**Limitación conocida:** SIPp con `-sf <path-absoluto>` escribe los archivos `_errors.log` y `_statistics.csv` en el **directorio del scenario** (`backend/sipp-scenarios/`), NO en el `cwd` que pasamos al spawn. Esto rompe el reader de CSV (`csvSummary parseado: null`). El backend cae al fallback de stderr snapshots, que funciona pero es menos preciso.

**TODO para el próximo dev:** mover el scenario XML al `sippCwd` antes de invocar SIPp, o usar flag `-error_file` y `-stat_file` (revisar docs SIPp 3.7.3).

### 4. Fix #3: stderr de SIPp sin filtro

**Problema:** el stderr filter (`/error|fatal|cannot|.../`) escondía mensajes útiles cuando SIPp moría sin matchear esos patrones (ej. el límite de sockets).

**Fix aplicado:** ([sippManager.js:170-173](../backend/src/services/sippManager.js#L170-L173))
```js
console.error('[SIPp:stderr]', line.trimEnd());  // todo crudo, sin filtro
```

Con esto encontramos el mensaje de límite de sockets de Cygwin que estaba escondido.

## Hallazgos del assessment (relevantes para el reporte OLAM)

### H-08: Anti-fraude del 3CX activo y funcionando ✅

**Evidencia:** después de varios `403 Invalid credentials` repetidos, el 3CX `.33` puso `172.18.164.35` en blacklist. Login web devolvió "An IP has been blacklisted".

**Implicación:** buena seguridad anti-toll-fraud. Para el reporte, esto es **un hallazgo positivo** — el sistema se protege automáticamente contra fuerza bruta SIP. Para tests de carga futuros, la IP del SBC tester debe whitelistearse en Security Settings antes de empezar.

**Cómo se desbloqueó:** Jose Bernal lo hizo manualmente desde el admin (5 minutos).

### H-09: Web login password ≠ SIP authentication password

**Evidencia:** `Lamda2026$04` autentica login web (https://regist2026.3cx.co/) pero el 3CX devuelve `403 Invalid credentials` cuando SIPp lo usa para auth digest.

**Implicación:** en 3CX hay dos campos separados:
- **Web login password** (configuración de la cuenta de usuario)
- **Authentication Password** (campo SIP, generado al provisionar un IP Phone)

El admin Jose dijo "los passwords son los mismos" pero se refería al web login. Para SIP necesitamos el campo separado, que solo existe cuando hay un IP Phone configurado en la extensión.

### H-10: Servers `.28` y `.33` son independientes (no clúster)

**Evidencia:** generamos credenciales SIP de la extensión 1910 en el `.28` (`d4eCqpsvYj`/`HpiO387MMi`) y las probamos contra `.33`. Resultado: `403 Invalid credentials`.

**Implicación:** el `.33` NO sincroniza extensiones desde el `.28`. Cada server tiene su propia base de usuarios. Para tests SIPp contra `.33` necesitamos credenciales generadas EN `.33`.

**Otro detalle:** el FQDN del Admin Console accesible vía `regist2026.3cx.co` (con "t") muestra `regis2026.3cx.co` (sin "t") y licencia SC32. Es decir, **el DNS `regist2026.3cx.co` redirige al `.28`** — el `.33` no tiene admin console accesible vía DNS público.

### H-11: Logs del 3CX en log level "Warning" — sin trazabilidad de llamadas

**Evidencia:** `tail -20` sobre `3CXCallFlow.log`, `3CXGatewayService.log`, `3cxSystemService.log` solo muestra líneas tipo "Log is started" y warnings de WebMeeting/Speech. No hay trazas de INVITEs ni de llamadas.

**Implicación:** las métricas en vivo de "llamadas activas" en el dashboard de OLAM **siempre van a estar en cero** mientras el log level esté en Warning (default productivo). El parser regex está bien — el problema es que no hay líneas para parsear.

**Para producción del dashboard:** pedir al admin OLAM cambiar el log level a "Info" en Management Console → Settings → Logs (durante ventanas de auditoría) para que llamadas individuales se loguen.

**Para el reporte:** esto NO es un blocker para el assessment. Las métricas de capacidad se obtienen vía:
- Output de SIPp (CSV + stderr)
- Métricas del host (CPU/RAM vía node_exporter local en `.35`)
- Manual SSH para revisar logs específicos cuando hace falta

### H-12: Files de output de SIPp en directorio del scenario

**Síntoma:** el reader del CSV no encuentra `_statistics.csv` en el `cwd` que el backend le pasa.
**Causa:** SIPp con `-sf <path-absoluto>` resuelve paths relativos al directorio del scenario, no al `cwd`. Los `errors.log` y `_statistics.csv` quedan en `backend/sipp-scenarios/uac_auth_*.csv`.
**Estado:** workaround funciona (fallback a snapshots de stderr). TODO listado en sección Fix #2.

## Cómo deployar a `.35`

Igual que en handoffs anteriores: zip + RDP folder redirection (`\\tsclient\C\...`).

**Desde la máquina dev (`C:\Users\Maximiliano Pulido\OLAM`):**

```powershell
# 1. Push a GitHub
git push origin main

# 2. Crear zip con archivos staging
$staging = "$env:TEMP\OLAM-deploy-staging"
if (Test-Path $staging) { Remove-Item $staging -Recurse -Force }
New-Item -ItemType Directory -Path $staging | Out-Null

$files = @(
  'backend/src/services/sippManager.js',
  'backend/sipp-scenarios/uac_auth.xml',
  'docs/HANDOFF-2026-05-05.md'
)

foreach ($f in $files) {
  $dest = Join-Path $staging $f
  $destDir = Split-Path $dest -Parent
  if (-not (Test-Path $destDir)) { New-Item -ItemType Directory -Path $destDir -Force | Out-Null }
  Copy-Item $f $dest
}

$zipPath = "$env:USERPROFILE\Desktop\OLAM-handoff-2026-05-05.zip"
if (Test-Path $zipPath) { Remove-Item $zipPath }
Compress-Archive -Path "$staging\*" -DestinationPath $zipPath -Force
```

