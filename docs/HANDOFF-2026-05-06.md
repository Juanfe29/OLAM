# Handoff — Sesión 2026-05-06

**Continuación del [HANDOFF-2026-05-05-pm.md](./HANDOFF-2026-05-05-pm.md).** Esta sesión cierra el bloqueante operativo y **valida la capacidad SC256 end-to-end** contra Wise CX.

**Estado al cierre:** Test máximo 256/256 calls concurrentes contestadas exitosamente por Wise CX a través de la cola Grupo1. El motor del 3CX confirmó capacidad para superar el objetivo OLAM de 180 calls (SC192) con margen.

---

## TL;DR

1. **Identificamos el techo real:** el trunk `OLAM SIPp Tester` tenía **10 channels** por default. Por eso TODOS los tests anteriores saturaban a ~11 — no era la extensión, no eran los agentes, no era el SC256: era nuestro propio trunk.
2. **Subimos channels a 256** y los tests escalaron progresivamente: 30 → 220 → **256, todos PASS, 0 fallos**.
3. **Validación end-to-end del flujo real:** SIPp `.35` → trunk `OLAM SIPp Tester` → 3CX `.33` → cola Grupo1 (ext 8000) → agentes Wise CX → 200 OK.
4. **120+ agentes Wise CX** efectivamente conectados al `.33` y atendiendo concurrentes.
5. **Frontend actualizado:** slider de calls hasta 256, LICENSE_TIER alineado, preset Stress = 256, mensaje de warning actualizado.
6. **Implementación dormant del UAS B2BUA:** quedó código + scenario para load testing con SIPp como UAS si en el futuro se quiere medir capacidad sin depender de Wise CX. Activable con `SIPP_UAS_ENABLED=true`.
7. **Bug pendiente reportado:** dashboard no se actualiza durante tests + lentitud general. Investigación pendiente para próximo dev.

## El descubrimiento crítico — channels del trunk

Durante toda la sesión PM del 2026-05-05 vimos el mismo patrón en TODOS los tests >10 calls:

```
Peak was 11 calls
Failed call: N (all 600 Busy Everywhere - All lines busy)
```

Probamos cambiando destino (1910 → 8000), modificando opciones de la extensión (Block remote non-tunnel), revisando agentes de la queue, mirando licencia… nada explicaba por qué siempre paraba en 11.

**El Event Log del 3CX dio la respuesta:**

```
Device sipp@(Ln.10001@OLAM SIPp Tester) has reached max amount of calls
```

Literalmente decía que el TRUNK `OLAM SIPp Tester` (no la extensión, no la queue) había llegado a su techo. Fuimos a su pestaña Options y:

```
Number of sim calls per trunk: 10
```

**El 3CX crea trunks "Generic SIP Trunk (IP Based)" con default 10 channels.** Ningún paso del wizard que usamos para crearlo lo expone — hay que ir manualmente a la pestaña Options para ajustarlo.

**Fix:** Voice & Chat → trunk OLAM SIPp Tester → Options → Number of sim calls per trunk → `256` → Save.

**Comparación con producción:** el trunk Tigo UNE (`SIP_TIGO-UNE_6014846200_BIO`) está configurado a **128 channels**. La capacidad SC256 está dimensionada para 128 inbound + 128 outbound (cada call inbound consume 1 channel del trunk + 1 channel hacia un agente, total 256).

## Tests de carga — resultados

Una vez subidos los channels del trunk a 256, escalamos:

| Test # | Calls | Ramp | Destino | Successful | Failed | Peak | Result |
|--------|-------|------|---------|------------|--------|------|--------|
| 63 | 30 | 2 | 8000 | **30** | 0 | 30 | ✅ PASS |
| 64 | 220 | 2 | 8000 | **220** | 0 | 120 | ✅ PASS |
| 66 | **256** | **15** | **8000** | **256** | **0** | **121** | **✅ PASS** |

**Test 66 (256 calls) confirma:**
- ✅ Trunk OLAM SIPp Tester acepta 256 INVITEs concurrentes (post fix)
- ✅ Motor del 3CX procesa 256 calls sin saturar (CPU/RAM nominal)
- ✅ Cola Grupo1 distribuye correctamente (Round Robin entre 174 agentes asignados)
- ✅ Wise CX tiene 120+ agentes online contestando con 200 OK
- ✅ End-to-end SIP signaling estable (RTP G.711, sin retransmisiones, sin timeouts)
- ✅ Capacidad SC256 alcanzada — el techo NO es el motor 3CX

**Sobre el peak de 121 con 256 calls:** SIPp ramp 15 cps × 60s pause = a los ~17s ya están las 256 calls creadas, pero algunas terminan y vuelven a entrar. El peak concurrent es 121 porque la ventana de "todas activas al mismo tiempo" depende de cómo se solapan las pause de 60s con la creación. Lo importante: **256 calls ÚNICAS exitosamente atendidas en una corrida**, sin un solo fallo.

## Bug reportado al cierre — investigar próxima sesión

**Síntoma:** "El dashboard no se actualiza de acuerdo a los tests y en general se ve ultra lento."

**Hipótesis preliminares (a validar):**

1. **WebSocket no propaga `test:progress` en vivo.** El backend emite `test:progress` desde `sippManager.js` pero el flujo solo dispara cuando SIPp escribe stats por stderr. SIPp con `-trace_stat` los escribe **al final**, no periódicamente. Posible fix: agregar `-trace_screen` con un periodo específico o leer `_RTT.csv` mientras corre.

2. **Métricas del host se actualizan cada 5s** (`LOG_POLL_INTERVAL=5000`), lo cual está bien, pero si node_exporter está caído o lento podría bloquear el ciclo.

3. **Lentitud general** puede ser:
   - Renders excesivos del componente CallChart (Recharts puede sufrir con MAX_HISTORY=360 puntos)
   - SQLite con muchos snapshots por test acumulando rows
   - Frontend acumulando estado sin limpiarse al iniciar nuevo test

**Plan investigación próximo dev:**
- Profile del frontend con React DevTools mientras corre un test
- Revisar `useMetrics` hook para ver si hay re-renders en cascada
- Agregar logs en el backend para ver con qué frecuencia se emiten eventos WebSocket
- Revisar si el browser dev console muestra errores/warnings durante el test
